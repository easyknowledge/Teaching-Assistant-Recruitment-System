## About The Project

Traditional Teaching Assistant (TA) recruitment processes at BUPT International School are often bogged down by scattered emails, opaque selection criteria, and inefficient communication between TAs, Module Organisers (MOs), and Administrators. 

**This project delivers a comprehensive system design and high-fidelity prototype** aimed at digitizing and streamlining the entire TA recruitment lifecycle—from application submission and role matching to workload tracking and approval workflows.

Developed as the capstone group project for **EBU6304 Software Engineering (Queen Mary University of London)**, this repository encapsulates the complete Software Engineering lifecycle:
- ✅ Initial requirements gathering (Document Analysis, Brainstorming, Interviewing)
- ✅ Agile Product Backlog planning (INVEST principle, MoSCoW prioritization, Story Points estimation)
- ✅ UI/UX prototyping (Role-based user journeys)
- ⏳ System architecture design (Upcoming for Intermediate Assessment)
- ⏳ Functional code implementation (Upcoming for Final Assessment)

### ✨ Core Features (Aligned with Product Backlog)
| Role | Key Features (Must-Have for MVP) |
|------|----------------------------------|
| 🧑🎓 TA Applicant | Account registration & login, role application, application status tracking, profile management |
| 👩🏫 Module Organiser (MO) | Role posting, applicant screening, TA assignment, workload approval |
| 👨💼 Administrator | System oversight, workload statistics, user management, role configuration |

### 🎯 Project Goals
- Reduce TA recruitment administrative overhead by 60%
- Ensure full transparency of application and selection processes
- Align with BUPT International School's academic workflow standards
- Adhere to Agile Scrum best practices and EBU6304 coursework requirements

## Collaboration Workflow

To satisfy the coursework requirements and keep contribution records clear:

* each member should work on a personal branch named as `{name}/{function}`
* each change should be committed and pushed through a pull request
* every pull request should be approved by at least one teammate before merge
* all meaningful progress should be visible on GitHub before each assessment checkpoint
